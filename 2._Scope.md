# Scope

The Open Component Model (OCM) specification is an open standard for describing software-bill-of-deliveries (SBOD). It provides a technology-agnostic, machine-readable format with a globally unique identity scheme for software artifacts throughout the entire software lifecycle management process.

Software artifacts described by OCM include any type of artifact involved in software delivery, including:

* Container images (OCI images, OCI artifacts)
* Package archives (Helm charts, npm packages)
* Binaries and executables
* File system content (directory trees, Git repositories)
* Software bill of materials (SBOMs)

This working group is focused specifically on establishing and promoting standards for the description, identification, signing, and transport of software artifacts. Its activities are focused on:

* A component model for describing versioned sets of software artifacts, including components, component versions, artifacts, and references ([Model](doc/01-model/README.md))
* Model element fundamentals, including identifiers, access specifications, labels, repository contexts, and signatures ([Model Elements](doc/01-model/03-elements-sub.md))
* Processing operations for referencing, signing, verification, and normalization of component versions ([Processing](doc/02-processing/README.md) specification)
* An abstract persistence layer, including model operations and storage backend mappings ([Persistence](doc/03-persistence/README.md) specification)
* An extension framework for artifact types, access method types, storage backend mappings, and algorithms ([Extensions](doc/04-extensions/README.md))
* Guidelines and conventions for transport, model contracts, and references ([Guidelines](doc/05-guidelines/README.md))

The working group's activities do not include:

* Building, compilation, or deployment of software artifacts. Such tasks are left to tools that operate on top of the model.
* Defining tool-specific metadata formats. Tool-specific metadata, such as deployment descriptions, is treated as typed artifacts within the model.
* Defining the internal format or structure of individual artifact types where existing standards are well established and can be reused (OCI image layout, Helm chart format).

Any changes of Scope are not retroactive.
